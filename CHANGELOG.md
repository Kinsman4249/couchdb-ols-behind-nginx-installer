## Changelog

This file tracks real changes to this repository. For the rules on how entries here should be written, see the changelog formatting guide in the templates repository.

### Initial installer (round one)
1. Created the repository to hold a native, no-Docker installer that stands up CouchDB as the backend for Obsidian Self-hosted LiveSync on Debian and wires it into an existing nginx reverse proxy. The installer is generic: every hostname, credential, token, path, and the TLS method are prompted for at runtime, and it is written to coexist with other vhosts already on the host rather than replace anything.
2. Added install-ols-couchdb.sh. The script re-executes itself under sudo when not run as root, then prompts for the hostname, CouchDB admin user and password, vault database name, any extra nginx include files to apply inside the vhost, and the TLS method. It enables the Apache CouchDB Debian repository, pre-seeds debconf so the package installs standalone and bound to localhost with an admin already set, installs CouchDB, runs the maintainer couchdb-init.sh to apply the CORS and require_valid_user settings the plugin needs, creates the vault database, installs the nginx rate-limit fragment, renders the SNI vhost by substituting the hostname, certificate paths, and any extra includes, and only enables the vhost once both the certificate and key files exist so that a config test cannot fail on a missing certificate path. If debconf keys differ on a given release, apt falls back to prompting and the printed values can be used to answer.
3. Added a TLS method menu covering certbot DNS-01 with a provider supplied by the user, certbot webroot, certbot standalone, bringing your own certificate by path, and skipping certificate acquisition entirely. The DNS-01 path prompts for the plugin package, the authenticator name, a credentials file path, and an email, and offers to create the credentials file interactively if it does not already exist, so any certbot-supported DNS provider works without changing the script.
4. Added the nginx SNI vhost template. It proxies to CouchDB on 127.0.0.1:5984, inserts any user-supplied include files at a marked location, sets HSTS scoped to the single host, sets client_max_body_size to zero so large notes and attachments are not truncated, and applies a generous burst rate limit as an anti-abuse net rather than a hard throttle. CORS is deliberately left to CouchDB and not duplicated in nginx, since duplicating it breaks replication.
5. Added the http-context rate-limit fragment as a separate file installed into conf.d, because the limit_req_zone directive must live in the http context and an externally managed nginx.conf should not be edited directly.
6. Added an optional 444 default_server drop for unknown-SNI and raw-IP connections on 443, offered as a prompt because a host may or may not already have a 443 default_server. It uses the snakeoil certificate so no real hostname or certificate is presented to scanners.
7. Added community-health files (Code of Conduct, Contributing, Security policy), a README with install-directly-from-GitHub instructions and the TLS option descriptions, an MIT license, and issue and pull request templates.

### Release tooling and Cloudflare guidance (round two)
8. Added a tag-driven GitHub Actions release workflow at .github/workflows/release.yml. It triggers on version tags matching v*.*.* and also offers a manual dispatch that takes an existing tag. Because this is a shell project with nothing to compile, the build job uses git archive to bundle the repository at the exact tagged commit into a .tar.gz and a .zip, uploads them as workflow artifacts, and a second job attaches those artifacts to a GitHub Release with auto-generated notes. The build matrix was trimmed to a single Linux runner since cross-platform compilation does not apply.
9. Added a README section describing how to place the endpoint behind a Cloudflare origin lockdown after installation using the CloudFlareDebianHardener project, including running the hardener first so its snippet files exist, pasting those snippet paths at the installer's extra-include prompt, and the order to follow if the hardener is run afterward instead. The section documents two interactions that matter: the hostname must stay proxied (orange) while the enforce snippet is active or every client receives 403, and certbot HTTP-01 must not be used for renewal under the lockdown because the validation servers are not Cloudflare IPs, so the DNS-01 option should be used instead.
